SAA-C03 — Domain 3: Design Secure Architectures 領域 3：設計安全架構

AWS Certified Solutions Architect – Associate  |  Deep-Dive Study Guide  |  30% of Exam AWS 認證解決方案架構師 – 助理  |  深度學習指南  |  占考試 30%

1. Domain Overview 領域概覽

Domain 3 tests your ability to design systems that protect data, control access, and detect threats using AWS security services. The exam focuses on choosing the right security control for the right layer — IAM for identity, KMS/encryption for data protection, security groups/NACLs/WAF for network, and GuardDuty/Inspector/Macie for detection. This domain rewards precision: knowing when one service applies and another doesn’t is what separates correct answers from plausible-looking traps.

領域 3 測試您使用 AWS 安全服務設計保護資料、控制存取和偵測威脅系統的能力。考試重點在於為正確的層選擇正確的安全控制——IAM 用於身分識別、KMS/加密用於資料保護、安全群組/NACL/WAF 用於網路，以及 GuardDuty/Inspector/Macie 用於偵測。此領域獎勵精準度：了解何時適用某個服務而另一個不適用，是區分正確答案和看似合理陷阱的關鍵。

2. Key AWS Services 主要 AWS 服務

Identity & Access Management (IAM) 身分識別和存取管理

IAM Policies IAM 政策

  • Identity-based policies: attached to users, groups, roles — define what that identity can do 以身分識別為基礎的政策：附加至使用者、群組、角色——定義該身分可以做什麼
  • Resource-based policies: attached to resources (S3 bucket policy, KMS key policy, Lambda resource policy) — define who can access that resource 以資源為基礎的政策：附加至資源（S3 儲存貯體政策、KMS 金鑰政策、Lambda 資源政策）——定義誰可以存取該資源
  • Permission boundaries: set the maximum permissions an identity can have — used to delegate administration safely 許可界限：設定身分可擁有的最大許可——用於安全地委派管理
  • Session policies: passed at role assumption; can only restrict (never expand) permissions 工作階段政策：在擔任角色時傳遞；只能限制（永遠不能擴展）許可
  • Evaluation order: explicit Deny → SCPs → Permission Boundaries → Identity policies → Resource policies 評估順序：明確拒絕 → SCP → 許可界限 → 以身分識別為基礎的政策 → 以資源為基礎的政策

IAM Roles IAM 角色

  • Temporary credentials via STS; no long-term access keys 透過 STS 的暫時憑證；無長期存取金鑰
  • EC2 instance profile: attach role to EC2 so app code uses role credentials — never hardcode keys EC2 執行個體設定檔：將角色附加至 EC2，讓應用程式程式碼使用角色憑證——永遠不要在程式碼中硬式編碼金鑰
  • Cross-account role: Account A assumes a role in Account B to access resources there 跨帳號角色：帳號 A 擔任帳號 B 中的角色以存取該處的資源
  • Service roles: allow AWS services (Lambda, ECS, etc.) to call other AWS services 服務角色：允許 AWS 服務（Lambda、ECS 等）呼叫其他 AWS 服務
  • Exam rule: “EC2 needs to access S3 securely” → IAM role + instance profile, never access keys in code 考試規則：「EC2 需要安全存取 S3」→ IAM 角色 + 執行個體設定檔，永遠不要在程式碼中使用存取金鑰

IAM Conditions IAM 條件

  • aws:MultiFactorAuthPresent — require MFA aws:MultiFactorAuthPresent —— 要求 MFA
  • aws:SourceIp — restrict access by IP aws:SourceIp —— 依 IP 限制存取
  • aws:RequestedRegion — restrict to specific regions aws:RequestedRegion —— 限制至特定區域
  • s3:prefix — restrict S3 access to specific object prefixes s3:prefix —— 限制 S3 存取至特定物件字首
  • Exam rule: “require MFA to delete S3 objects” → IAM policy with aws:MultiFactorAuthPresent condition 考試規則：「要求 MFA 才能刪除 S3 物件」→ 具有 aws:MultiFactorAuthPresent 條件的 IAM 政策

AWS Organizations & SCPs AWS Organizations 與服務控制政策

  • SCPs (Service Control Policies): guardrails applied to accounts or OUs — define the maximum allowed permissions across the org SCP（服務控制政策）：套用至帳號或組織單位的護欄——定義整個組織允許的最大許可
  • SCPs do NOT grant permissions — they only restrict what IAM can grant SCP 不授予許可——它們只限制 IAM 可以授予的內容
  • SCPs apply to all principals in member accounts, except the management (root) account SCP 套用至成員帳號中的所有主體，管理（根）帳號除外
  • Exam rule: “prevent any IAM user from disabling CloudTrail across all accounts” → SCP, not IAM policy 考試規則：「防止任何 IAM 使用者在所有帳號中停用 CloudTrail」→ SCP，而非 IAM 政策

IAM Access Analyzer IAM 存取分析器

  • Identifies resources shared with external entities (outside the account or org) 識別與外部實體（帳號或組織外部）共用的資源
  • Exam rule: “find S3 buckets or IAM roles accessible from outside the organization” → IAM Access Analyzer 考試規則：「尋找可從組織外部存取的 S3 儲存貯體或 IAM 角色」→ IAM 存取分析器

S3 Security S3 安全

Access Control Layers 存取控制層

  • Block Public Access: account-level or bucket-level; overrides ACLs and bucket policies; must be disabled to allow any public access 封鎖公有存取：帳號層級或儲存貯體層級；覆寫 ACL 和儲存貯體政策；必須停用才能允許任何公有存取
  • Bucket Policies: JSON resource-based policies; grant access to other accounts or enforce conditions (TLS, VPC endpoint) 儲存貯體政策：JSON 以資源為基礎的政策；授予其他帳號存取權或強制執行條件（TLS、VPC 端點）
  • ACLs: legacy object-level control; AWS recommends leaving them disabled in modern setups ACL：舊版物件層級控制；AWS 建議在現代設定中保持停用
  • S3 Access Points: named endpoints with own access policies; simplify access for shared datasets with multiple teams S3 存取點：具有自己存取政策的具名端點；簡化多個團隊共用資料集的存取

S3 Encryption S3 加密

Type 類型 Key Management 金鑰管理 Use Case 使用案例
SSE-S3 (AES-256)AWS manages entirely AWS 完全管理Default, lowest overhead 預設、最低負荷
SSE-KMSAWS KMS manages CMK AWS KMS 管理 CMKAudit trail, key control, cross-account 稽核追蹤、金鑰控制、跨帳號
SSE-CCustomer provides key per request 客戶每次請求提供金鑰Customer retains full key control 客戶保留完整金鑰控制

Exam rule: “need audit trail of key usage” → SSE-KMS; “AWS must not access key material” → SSE-C or client-side; “default encryption with no overhead” → SSE-S3

考試規則：「需要金鑰使用的稽核追蹤」→ SSE-KMS；「AWS 不得存取金鑰材料」→ SSE-C 或用戶端；「無負荷的預設加密」→ SSE-S3

Presigned URLs 預先簽署 URL

  • Temporary access to a private S3 object without changing bucket permissions 在不變更儲存貯體許可的情況下臨時存取私有 S3 物件
  • Generated by any principal with access; expires after a configured time (max 7 days via STS credentials) 由任何具有存取權的主體產生；在設定的時間後到期（透過 STS 憑證最長 7 天）
  • Exam rule: “share a private S3 file temporarily without making it public” → presigned URL 考試規則：「臨時共用私有 S3 檔案而不公開」→ 預先簽署 URL

S3 MFA Delete S3 MFA 刪除

  • Requires MFA to permanently delete objects or suspend versioning 需要 MFA 才能永久刪除物件或暫停版本控制
  • Can only be enabled by the root account user (not IAM users) 只能由根帳號使用者啟用（非 IAM 使用者）
  • Exam rule: “protect against accidental or malicious deletion” → versioning + MFA Delete 考試規則：「防止意外或惡意刪除」→ 版本控制 + MFA 刪除

Encryption & Key Management 加密與金鑰管理

AWS KMS AWS 金鑰管理服務

  • Managed encryption key service; keys never leave KMS unencrypted 受管加密金鑰服務；金鑰絕不會以未加密狀態離開 KMS
  • AWS-managed CMK: free, no direct control; Customer-managed CMK: full control, key policies, rotation; AWS-owned: used internally by services AWS 受管 CMK：免費、無直接控制；客戶受管 CMK：完整控制、金鑰政策、輪換；AWS 擁有：服務內部使用
  • Key policies: resource-based policies that control who can use/manage the key — must exist for any principal to use the key 金鑰政策：控制誰可以使用/管理金鑰的以資源為基礎的政策——任何主體使用金鑰都必須存在金鑰政策
  • Envelope encryption: plaintext data encrypted with a data key; data key encrypted with CMK; only encrypted data key is stored 信封加密：純文字資料用資料金鑰加密；資料金鑰用 CMK 加密；僅儲存已加密的資料金鑰
  • Automatic key rotation: optional; rotates key material annually; old key retained to decrypt existing data 自動金鑰輪換：選用；每年輪換金鑰材料；保留舊金鑰以解密現有資料
  • KMS keys are regional; use multi-region keys for cross-region scenarios KMS 金鑰為區域性；跨區域場景使用多區域金鑰

CloudHSM

  • Dedicated hardware security module in your VPC VPC 中的專用硬體安全模組
  • FIPS 140-2 Level 3 certified (KMS is Level 2) FIPS 140-2 Level 3 認證（KMS 為 Level 2）
  • AWS has zero access to keys — customer-managed entirely AWS 對金鑰的存取權為零——完全由客戶管理
  • Exam rule: “FIPS 140-2 Level 3 required” or “AWS must have zero access to key material” → CloudHSM; otherwise → KMS 考試規則：「需要 FIPS 140-2 Level 3」或「AWS 必須對金鑰材料的存取權為零」→ CloudHSM；否則 → KMS

ACM (AWS Certificate Manager) ACM（AWS 憑證管理員）

  • Provisions, manages, and renews SSL/TLS certificates 佈建、管理和更新 SSL/TLS 憑證
  • Free for certificates used with ALB, CloudFront, API Gateway 與 ALB、CloudFront、API Gateway 搭配使用的憑證免費
  • Certificates are NOT exportable from ACM for use outside AWS services 憑證不可從 ACM 匯出供 AWS 服務以外使用
  • Exam rule: “HTTPS on ALB or CloudFront” → ACM; “certificate needed on EC2 directly” → ACM Private CA or third-party 考試規則：「ALB 或 CloudFront 上的 HTTPS」→ ACM；「EC2 上直接需要憑證」→ ACM Private CA 或第三方

Secrets Management 秘密管理

AWS Secrets Manager AWS Secrets Manager

  • Stores and automatically rotates secrets (DB credentials, API keys) 儲存並自動輪換秘密（資料庫憑證、API 金鑰）
  • Native integration with RDS, Redshift, DocumentDB for rotation 與 RDS、Redshift、DocumentDB 原生整合以進行輪換
  • Cost: ~$0.40/secret/month 費用：約 $0.40/秘密/月
  • Exam rule: “automatically rotate database credentials” → Secrets Manager 考試規則：「自動輪換資料庫憑證」→ Secrets Manager

Systems Manager Parameter Store Systems Manager 參數存放區

  • Stores configuration and secrets as name/value pairs 以名稱/值配對方式儲存設定和秘密
  • Standard tier: free, max 4 KB; Advanced tier: max 8 KB, supports parameter policies 標準層：免費、最大 4 KB；進階層：最大 8 KB、支援參數政策
  • SecureString uses KMS encryption SecureString 使用 KMS 加密
  • Exam rule: “store config and secrets with no auto-rotation needed” → Parameter Store (cheaper); “auto-rotation required” → Secrets Manager 考試規則：「儲存不需要自動輪換的設定和秘密」→ 參數存放區（較便宜）；「需要自動輪換」→ Secrets Manager

Network Security 網路安全

Security Groups 安全群組

  • Stateful: return traffic automatically allowed 具狀態：傳回流量自動允許
  • Applied at the ENI (instance) level; allow rules only (no explicit deny) 套用於 ENI（執行個體）層級；僅限允許規則（無明確拒絕）
  • Multiple security groups can be applied; rules are unioned 可套用多個安全群組；規則取聯集
  • Exam rule: “block a specific IP from reaching an instance” → Security Groups cannot block by IP; use NACL instead 考試規則：「封鎖特定 IP 到達執行個體」→ 安全群組無法依 IP 封鎖；改用 NACL

Network ACLs (NACLs) 網路存取控制清單

  • Stateless: must explicitly allow both inbound AND outbound (including ephemeral ports 1024–65535) 無狀態：必須明確允許輸入和輸出（包括暫時連接埠 1024–65535）
  • Applied at the subnet level; support both allow and deny rules 套用於子網路層級；支援允許和拒絕規則
  • Rules evaluated in numbered order, lowest first; first match wins 規則依編號順序評估，最小號碼優先；第一個相符規則勝出
  • Exam rule: “block a specific IP range at the subnet level” → NACL; “instance-level filtering” → Security Group 考試規則：「在子網路層級封鎖特定 IP 範圍」→ NACL；「執行個體層級篩選」→ 安全群組

VPC Flow Logs VPC 流量日誌

  • Capture metadata about IP traffic to/from ENIs, subnets, or VPCs; logs to CloudWatch or S3 擷取往返 ENI、子網路或 VPC 的 IP 流量中繼資料；日誌記錄至 CloudWatch 或 S3
  • Does NOT capture packet content (use traffic mirroring for that) 不擷取封包內容（使用流量鏡像來實現）
  • Exam rule: “investigate which IPs are connecting to EC2 instances” → VPC Flow Logs 考試規則：「調查哪些 IP 正在連接 EC2 執行個體」→ VPC 流量日誌

NAT Gateway NAT 閘道

  • Allows private subnet instances to initiate outbound internet traffic; blocks inbound 允許私有子網路執行個體發起輸出網際網路流量；封鎖輸入
  • Managed by AWS, HA within an AZ; deploy one per AZ for full HA 由 AWS 管理，在可用區域內具有高可用性；每個可用區域部署一個以實現完整高可用性
  • Exam rule: “private EC2 needs to download patches from the internet” → NAT Gateway 考試規則：「私有 EC2 需要從網際網路下載修補程式」→ NAT 閘道

AWS WAF AWS Web 應用程式防火牆

  • Layer 7 protection; inspects HTTP/HTTPS requests; protects against SQL injection, XSS, bad bots 第 7 層保護；檢查 HTTP/HTTPS 請求；防護 SQL 注入、XSS、惡意機器人
  • Deployed on ALB, CloudFront, API Gateway, AppSync 部署於 ALB、CloudFront、API Gateway、AppSync
  • Rate-based rules: block IPs exceeding a threshold (Layer 7 DDoS mitigation) 以速率為基礎的規則：封鎖超過閾值的 IP（第 7 層 DDoS 緩解）
  • Exam rule: “protect web app from SQL injection and XSS” → WAF; “Layer 3/4 DDoS” → Shield 考試規則：「保護 Web 應用程式免受 SQL 注入和 XSS 攻擊」→ WAF；「第 3/4 層 DDoS」→ Shield

AWS Shield AWS Shield

  • Shield Standard: automatic Layer 3/4 DDoS protection; free for all customers Shield 標準版：自動第 3/4 層 DDoS 防護；所有客戶免費
  • Shield Advanced: enhanced protection, 24/7 DRT access, cost protection, near-real-time visibility; ~$3,000/month Shield 進階版：增強防護、24/7 DRT 存取、費用保護、接近即時可見性；約 $3,000/月
  • Exam rule: “DDoS response team support and cost protection” → Shield Advanced; basic protection → Standard (already included) 考試規則：「DDoS 回應團隊支援和費用保護」→ Shield 進階版；基本防護 → 標準版（已包含）

AWS Network Firewall AWS 網路防火牆

  • Managed stateful firewall in VPC; deep packet inspection; Suricata-based rules VPC 中的受管具狀態防火牆；深度封包檢查；以 Suricata 為基礎的規則
  • Exam rule: “intrusion prevention”, “deep packet inspection at VPC level” → Network Firewall; web traffic filtering → WAF 考試規則：「入侵防護」、「VPC 層級的深度封包檢查」→ 網路防火牆；Web 流量篩選 → WAF

Threat Detection & Monitoring 威脅偵測與監控

Amazon GuardDuty Amazon GuardDuty（守衛職責）

  • Intelligent threat detection using ML; analyzes CloudTrail, VPC Flow Logs, DNS logs, S3 data events 使用 ML 的智慧型威脅偵測；分析 CloudTrail、VPC 流量日誌、DNS 日誌、S3 資料事件
  • Detects: compromised credentials, unusual API calls, port scanning, crypto-mining, data exfiltration 偵測：遭入侵的憑證、異常 API 呼叫、連接埠掃描、加密貨幣挖掘、資料竊取
  • No agents required; one-click enable 無需代理程式；一鍵啟用
  • Exam rule: “detect threats and suspicious activity across the account” → GuardDuty 考試規則：「偵測帳號中的威脅和可疑活動」→ GuardDuty

Amazon Inspector Amazon Inspector（巡查員）

  • Automated vulnerability assessment for EC2, containers (ECR), and Lambda EC2、容器（ECR）和 Lambda 的自動化弱點評估
  • Finds OS vulnerabilities, network exposure, unintended software 尋找作業系統弱點、網路暴露、非預期軟體
  • Exam rule: “scan EC2 for known vulnerabilities and CVEs” → Inspector; “detect active threats” → GuardDuty 考試規則：「掃描 EC2 的已知弱點和 CVE」→ Inspector；「偵測主動威脅」→ GuardDuty

Amazon Macie Amazon Macie（資料安全服務）

  • Uses ML to discover and protect sensitive data (PII, financial data, credentials) in S3 使用 ML 在 S3 中探索和保護敏感資料（個人識別資訊、財務資料、憑證）
  • Exam rule: “identify S3 buckets containing PII” → Macie 考試規則：「識別包含個人識別資訊的 S3 儲存貯體」→ Macie

AWS Security Hub AWS Security Hub（安全中樞）

  • Centralized view of security findings across GuardDuty, Inspector, Macie, WAF, Access Analyzer GuardDuty、Inspector、Macie、WAF、存取分析器的安全調查結果集中檢視
  • Aggregates across accounts using AWS Organizations 使用 AWS Organizations 跨帳號彙總
  • Exam rule: “single pane of glass for security findings across multiple accounts” → Security Hub 考試規則：「跨多個帳號安全調查結果的單一管理介面」→ Security Hub

AWS CloudTrail AWS CloudTrail（雲端追蹤）

  • Records every API call: who did what, when, from where 記錄每個 API 呼叫：誰做了什麼、何時、從哪裡
  • Management events: control plane (CreateBucket, RunInstances) — enabled by default 管理事件：控制平面（CreateBucket、RunInstances）——預設啟用
  • Data events: S3 object-level and Lambda invocations — off by default, costs extra 資料事件：S3 物件層級和 Lambda 叫用——預設關閉，需額外付費
  • CloudTrail Insights: detects unusual API activity patterns CloudTrail Insights：偵測異常 API 活動模式
  • Exam rule: “audit who deleted the S3 bucket” → CloudTrail; “real-time alerts on API calls” → CloudTrail + CloudWatch Events 考試規則：「稽核誰刪除了 S3 儲存貯體」→ CloudTrail；「API 呼叫的即時警示」→ CloudTrail + CloudWatch Events

AWS Config AWS Config（設定服務）

  • Tracks configuration changes to AWS resources; evaluates against compliance rules 追蹤 AWS 資源的設定變更；根據合規規則評估
  • CloudTrail = who did what; Config = what does the resource look like now/then CloudTrail = 誰做了什麼；Config = 資源現在/當時是什麼樣子
  • Exam rule: “check if all S3 buckets have encryption enabled” or “track configuration drift” → AWS Config 考試規則：「檢查所有 S3 儲存貯體是否啟用加密」或「追蹤設定漂移」→ AWS Config

Identity Federation & Cognito 身分識別聯合與 Cognito

Cognito User Pools Cognito 使用者集區

  • User directory for web/mobile apps; handles sign-up, sign-in, MFA, password recovery Web/行動應用程式的使用者目錄；處理註冊、登入、MFA、密碼復原
  • Issues JWT tokens (ID token, access token, refresh token) 發出 JWT 權杖（ID 權杖、存取權杖、重新整理權杖）
  • Exam rule: “add user authentication to a mobile app” → Cognito User Pools 考試規則：「將使用者驗證新增至行動應用程式」→ Cognito 使用者集區

Cognito Identity Pools (Federated Identities) Cognito 身分識別集區（聯合身分識別）

  • Provides temporary AWS credentials to users; works with User Pools, social providers, SAML 向使用者提供暫時 AWS 憑證；與使用者集區、社群媒體提供者、SAML 搭配使用
  • Exam rule: “allow mobile app users to directly access S3 or DynamoDB” → Cognito Identity Pools 考試規則：「允許行動應用程式使用者直接存取 S3 或 DynamoDB」→ Cognito 身分識別集區

IAM Identity Center (SSO) IAM Identity Center（單一登入）

  • Centralized SSO for multiple AWS accounts and business applications 多個 AWS 帳號和商業應用程式的集中式 SSO
  • Integrates with Active Directory, Okta, Azure AD 與 Active Directory、Okta、Azure AD 整合
  • Exam rule: “single sign-on for employees across multiple AWS accounts” → IAM Identity Center 考試規則：「員工跨多個 AWS 帳號的單一登入」→ IAM Identity Center

3. Architecture Patterns 架構模式

Pattern 1: Secure Multi-Tier Web Application 模式 1：安全的多層 Web 應用程式

Services: CloudFront (WAF) → ALB (Security Group) → EC2 in private subnets → RDS in isolated subnets

服務：CloudFront（WAF）→ ALB（安全群組）→ 私有子網路中的 EC2 → 隔離子網路中的 RDS

Why it works: WAF filters malicious traffic at edge. ALB security group allows only HTTPS from CloudFront IPs. EC2 security group allows only traffic from ALB. RDS security group allows only traffic from EC2. No direct internet exposure to app or DB tier.

為何有效：WAF 在邊緣篩選惡意流量。ALB 安全群組僅允許來自 CloudFront IP 的 HTTPS。EC2 安全群組僅允許來自 ALB 的流量。RDS 安全群組僅允許來自 EC2 的流量。應用程式層或資料庫層不直接暴露於網際網路。

Exam signal: “web application with defense in depth” → CloudFront + WAF + layered security groups, no public IPs on app/DB tier.

考試信號：「具深度防禦的 Web 應用程式」→ CloudFront + WAF + 分層安全群組，應用程式/資料庫層無公有 IP。

Pattern 2: Secure S3 Data Lake 模式 2：安全的 S3 資料湖

Services: S3 (SSE-KMS, Block Public Access ON) + Bucket Policy (enforce TLS + VPC endpoint) + Macie + CloudTrail data events

服務：S3（SSE-KMS、封鎖公有存取開啟）+ 儲存貯體政策（強制 TLS + VPC 端點）+ Macie + CloudTrail 資料事件

Why it works: SSE-KMS encrypts with auditable key usage. Block Public Access prevents accidental exposure. Bucket policy denies non-TLS requests. Macie monitors for sensitive data. CloudTrail data events capture every object-level access.

為何有效：SSE-KMS 使用可稽核金鑰使用進行加密。封鎖公有存取防止意外暴露。儲存貯體政策拒絕非 TLS 請求。Macie 監控敏感資料。CloudTrail 資料事件擷取每個物件層級存取。

Exam signal: “sensitive data in S3 with full audit trail and encryption” → SSE-KMS + CloudTrail data events + Macie.

考試信號：「S3 中具完整稽核追蹤和加密的敏感資料」→ SSE-KMS + CloudTrail 資料事件 + Macie。

Pattern 3: Cross-Account Access 模式 3：跨帳號存取

Services: IAM Role in Account B (trust policy allows Account A) → Account A IAM principal assumes the role via STS

服務：帳號 B 中的 IAM 角色（信任政策允許帳號 A）→ 帳號 A 的 IAM 主體透過 STS 擔任角色

Why it works: No long-term credentials cross account boundaries. Permissions scoped to the role in Account B. Fully audited via CloudTrail.

為何有效：無長期憑證跨帳號邊界。許可範圍限定於帳號 B 中的角色。透過 CloudTrail 完整稽核。

Exam signal: “application in Account A needs to read DynamoDB in Account B” → create role in Account B, trust Account A, assume it from Account A.

考試信號：「帳號 A 中的應用程式需要讀取帳號 B 中的 DynamoDB」→ 在帳號 B 中建立角色、信任帳號 A、從帳號 A 擔任角色。

Pattern 4: Secrets Rotation for RDS 模式 4：RDS 的秘密輪換

Services: RDS → Secrets Manager (automatic rotation Lambda) → Application retrieves secret at runtime

服務：RDS → Secrets Manager（自動輪換 Lambda）→ 應用程式在執行時期擷取秘密

Why it works: Secrets Manager rotates credentials on schedule without manual intervention. Application never has hardcoded credentials.

為何有效：Secrets Manager 按排程輪換憑證，無需手動干預。應用程式永遠不會有硬式編碼的憑證。

Exam signal: “automatically rotate database passwords without application changes” → Secrets Manager with RDS integration.

考試信號：「在不變更應用程式的情況下自動輪換資料庫密碼」→ 具有 RDS 整合的 Secrets Manager。

Pattern 5: Centralized Security Monitoring 模式 5：集中式安全監控

Services: GuardDuty + Security Hub + CloudTrail + Config → EventBridge → SNS (alerts) or Lambda (automated remediation)

服務：GuardDuty + Security Hub + CloudTrail + Config → EventBridge → SNS（警示）或 Lambda（自動修復）

Why it works: GuardDuty detects threats; Config tracks compliance drift; CloudTrail provides audit logs; Security Hub aggregates findings; EventBridge triggers automated responses.

為何有效：GuardDuty 偵測威脅；Config 追蹤合規漂移；CloudTrail 提供稽核日誌；Security Hub 彙總調查結果；EventBridge 觸發自動回應。

Exam signal: “detect and automatically remediate security issues across all accounts” → GuardDuty + Security Hub + EventBridge + Lambda.

考試信號：「偵測並自動修復所有帳號的安全問題」→ GuardDuty + Security Hub + EventBridge + Lambda。

4. Service Comparisons 服務比較

Security Groups vs. NACLs 安全群組 vs. NACL

  • Security Groups = stateful, instance level, allow rules only 安全群組 = 具狀態、執行個體層級、僅限允許規則
  • NACLs = stateless, subnet level, allow and deny, numbered rules (lowest first, first match wins) NACL = 無狀態、子網路層級、允許和拒絕、編號規則（最小號碼優先，第一個相符勝出）
  • Exam rule: “block a specific IP address” → NACL (Security Groups can’t deny); “control traffic between instances in same subnet” → Security Groups 考試規則：「封鎖特定 IP 位址」→ NACL（安全群組無法拒絕）；「控制同一子網路中執行個體間的流量」→ 安全群組

KMS vs. CloudHSM

  • KMS = managed, shared HSM, FIPS 140-2 Level 2, AWS has limited access KMS = 受管、共用 HSM、FIPS 140-2 Level 2、AWS 有限存取
  • CloudHSM = dedicated hardware, FIPS 140-2 Level 3, zero AWS access CloudHSM = 專用硬體、FIPS 140-2 Level 3、AWS 存取權為零
  • Exam rule: “FIPS 140-2 Level 3” or “AWS must have no access to key material” → CloudHSM; all other encryption → KMS 考試規則：「FIPS 140-2 Level 3」或「AWS 不得對金鑰材料有任何存取權」→ CloudHSM；所有其他加密 → KMS

Secrets Manager vs. Parameter Store Secrets Manager vs. 參數存放區

  • Secrets Manager = auto-rotation, RDS integration, ~$0.40/secret/month Secrets Manager = 自動輪換、RDS 整合、約 $0.40/秘密/月
  • Parameter Store = free for standard, no auto-rotation, good for config + secrets 參數存放區 = 標準層免費、無自動輪換、適合設定 + 秘密
  • Exam rule: “automatic credential rotation” → Secrets Manager; “store app config cheaply” → Parameter Store 考試規則：「自動憑證輪換」→ Secrets Manager；「便宜儲存應用程式設定」→ 參數存放區

GuardDuty vs. Inspector vs. Macie

  • GuardDuty = active threat detection (runtime, network, API behavior) — no agents GuardDuty = 主動威脅偵測（執行時期、網路、API 行為）——無代理程式
  • Inspector = vulnerability scanning (CVEs, misconfigurations in EC2/containers) Inspector = 弱點掃描（CVE、EC2/容器中的設定錯誤）
  • Macie = sensitive data discovery in S3 (PII, financial data) Macie = S3 中的敏感資料探索（個人識別資訊、財務資料）
  • Exam rule: “detect compromised instance behavior” → GuardDuty; “find unpatched OS vulnerabilities” → Inspector; “find PII in S3” → Macie 考試規則：「偵測遭入侵的執行個體行為」→ GuardDuty；「尋找未修補的作業系統弱點」→ Inspector；「在 S3 中尋找個人識別資訊」→ Macie

Shield Standard vs. Advanced Shield 標準版 vs. 進階版

  • Standard = free, automatic, Layer 3/4 DDoS protection 標準版 = 免費、自動、第 3/4 層 DDoS 防護
  • Advanced = ~$3K/month, Layer 7, DRT access, cost protection, real-time metrics 進階版 = 約 $3,000/月、第 7 層、DRT 存取、費用保護、即時指標
  • Exam rule: “DRT support and financial protection from DDoS” → Shield Advanced; basic protection → Standard (already included) 考試規則：「DRT 支援和 DDoS 財務保護」→ Shield 進階版；基本防護 → 標準版（已包含）

Cognito User Pools vs. Identity Pools Cognito 使用者集區 vs. 身分識別集區

  • User Pools = authentication (sign-in, JWT tokens, user directory) 使用者集區 = 驗證（登入、JWT 權杖、使用者目錄）
  • Identity Pools = AWS credentials (allow users to call AWS APIs directly) 身分識別集區 = AWS 憑證（允許使用者直接呼叫 AWS API）
  • Exam rule: “app users need to sign in” → User Pools; “app users need AWS credentials to access S3/DynamoDB” → Identity Pools; both needed → use together 考試規則：「應用程式使用者需要登入」→ 使用者集區；「應用程式使用者需要 AWS 憑證存取 S3/DynamoDB」→ 身分識別集區；兩者皆需 → 一起使用

SCPs vs. IAM Policies SCP vs. IAM 政策

  • SCPs = org-wide guardrails; define the ceiling of permissions; do NOT grant permissions SCP = 整個組織的護欄；定義許可的上限；不授予許可
  • IAM Policies = grant/deny permissions within an account IAM 政策 = 在帳號內授予/拒絕許可
  • Exam rule: “prevent all accounts in the org from leaving Organizations” → SCP; “allow a user to access S3” → IAM policy 考試規則：「防止組織中的所有帳號離開 Organizations」→ SCP；「允許使用者存取 S3」→ IAM 政策

5. Exam Traps & Gotchas 考試陷阱與注意事項

  1. Security Groups cannot deny traffic. To explicitly block an IP, use a NACL. The exam will ask “block a specific malicious IP” — the answer is always NACL. 安全群組無法拒絕流量。要明確封鎖 IP，使用 NACL。考試會問「封鎖特定惡意 IP」——答案始終是 NACL。
  2. NACLs are stateless — remember ephemeral ports. When allowing inbound port 443, you must also allow outbound ephemeral ports (1024–65535) for the response. Forgetting this causes connectivity failures in NACL scenarios. NACL 是無狀態的——記住暫時連接埠。允許輸入連接埠 443 時，也必須允許輸出暫時連接埠（1024–65535）以供回應。忘記這點會在 NACL 場景中導致連線失敗。
  3. SCPs don’t grant permissions; they only restrict. Even if an SCP allows everything, an IAM policy must still explicitly grant access. A user can’t access S3 with only an SCP — they need an IAM policy too. SCP 不授予許可；它們只限制。即使 SCP 允許所有內容，IAM 政策仍必須明確授予存取權。使用者僅憑 SCP 無法存取 S3——他們還需要 IAM 政策。
  4. IAM role trust policy vs. permission policy. A role has two policies: the trust policy (who can assume the role) and the permission policy (what the role can do). Cross-account failures often stem from a missing trust policy, not missing permissions. IAM 角色信任政策 vs. 許可政策。角色有兩個政策：信任政策（誰可以擔任角色）和許可政策（角色可以做什麼）。跨帳號失敗通常源於缺少信任政策，而非缺少許可。
  5. SSE-KMS has API call costs. Every S3 GET/PUT with SSE-KMS results in a KMS API call. High-throughput S3 workloads can hit KMS limits. If audit trail is not required → SSE-S3 may be more appropriate. SSE-KMS 有 API 呼叫成本。每次使用 SSE-KMS 的 S3 GET/PUT 都會產生 KMS API 呼叫。高輸送量 S3 工作負載可能達到 KMS 限制。若不需要稽核追蹤 → SSE-S3 可能更合適。
  6. CloudTrail data events are off by default. Management events (CreateBucket, RunInstances) log by default. S3 object-level access (GetObject, PutObject) must be explicitly enabled and costs extra. CloudTrail 資料事件預設關閉。管理事件（CreateBucket、RunInstances）預設記錄。S3 物件層級存取（GetObject、PutObject）必須明確啟用並需額外付費。
  7. GuardDuty doesn’t block traffic. It only detects and alerts. Automated remediation requires EventBridge + Lambda. GuardDuty itself takes no blocking action. GuardDuty 不封鎖流量。它只偵測和發出警示。自動修復需要 EventBridge + Lambda。GuardDuty 本身不採取任何封鎖動作。
  8. Cognito User Pools vs. Identity Pools confusion. JWTs from User Pools cannot authenticate directly to AWS APIs — they require Identity Pools to exchange for AWS credentials. If the scenario involves calling S3 or DynamoDB from a mobile app, Identity Pools are required. Cognito 使用者集區 vs. 身分識別集區混淆。來自使用者集區的 JWT 無法直接向 AWS API 驗證——它們需要身分識別集區兌換為 AWS 憑證。若場景涉及從行動應用程式呼叫 S3 或 DynamoDB，則需要身分識別集區。

7. Practice Questions 練習題

Question 1 — A security team discovers that an EC2 instance is making unusual API calls and connecting to known malicious IP addresses. The team wants a service that can automatically detect this behavior across all AWS accounts in the organization without installing agents. Which service should the solutions architect enable? 問題 1——安全團隊發現 EC2 執行個體正在進行異常 API 呼叫並連接至已知的惡意 IP 位址。團隊希望有一個服務能自動偵測整個組織所有 AWS 帳號中的此類行為，且無需安裝代理程式。解決方案架構師應啟用哪個服務？
  1. Amazon Inspector Amazon Inspector
  2. AWS Config AWS Config
  3. Amazon GuardDuty Amazon GuardDuty
  4. AWS Security Hub AWS Security Hub
Answer: C 答案：C
  • C is correct: GuardDuty uses ML to analyze CloudTrail logs, VPC Flow Logs, and DNS logs to detect threats like compromised instances and malicious IP connections. No agents needed; works across the organization. C 正確：GuardDuty 使用 ML 分析 CloudTrail 日誌、VPC 流量日誌和 DNS 日誌，以偵測遭入侵執行個體和惡意 IP 連線等威脅。無需代理程式；在整個組織中運作。
  • A is wrong: Inspector scans for vulnerabilities and misconfigurations, not active runtime threats or network behavior. A 錯誤：Inspector 掃描弱點和設定錯誤，而非主動執行時期威脅或網路行為。
  • B is wrong: Config tracks resource configuration changes for compliance, not active threat detection. B 錯誤：Config 追蹤資源設定變更以符合合規，而非主動威脅偵測。
  • D is wrong: Security Hub aggregates findings from GuardDuty and others but does not detect threats on its own. D 錯誤：Security Hub 彙總來自 GuardDuty 和其他服務的調查結果，但本身不偵測威脅。
Question 2 — A company stores sensitive customer PII in S3. They need to ensure all objects are encrypted with a key they control, all key usage is audited, and objects cannot be uploaded without encryption. What should the solutions architect configure? 問題 2——某公司在 S3 中儲存敏感的客戶個人識別資訊。他們需要確保所有物件都使用他們控制的金鑰加密、所有金鑰使用都受到稽核，且物件不得在未加密的情況下上傳。解決方案架構師應設定什麼？
  1. Enable SSE-S3 as default encryption and add a bucket policy to deny unencrypted uploads 啟用 SSE-S3 作為預設加密，並新增儲存貯體政策以拒絕未加密的上傳
  2. Enable SSE-KMS with a customer-managed CMK, add a bucket policy denying uploads without the KMS header, and enable CloudTrail 使用客戶受管 CMK 啟用 SSE-KMS，新增拒絕不含 KMS 標頭上傳的儲存貯體政策，並啟用 CloudTrail
  3. Use client-side encryption and upload pre-encrypted objects to S3 使用用戶端加密並將預先加密的物件上傳至 S3
  4. Enable SSE-C and require the client to provide the encryption key on every request 啟用 SSE-C 並要求用戶端在每次請求時提供加密金鑰
Answer: B 答案：B
  • B is correct: SSE-KMS with a customer-managed CMK gives full key control. CloudTrail logs every KMS API call. The bucket policy enforces encryption on all uploads. Meets all three requirements. B 正確：具有客戶受管 CMK 的 SSE-KMS 提供完整金鑰控制。CloudTrail 記錄每個 KMS API 呼叫。儲存貯體政策對所有上傳強制執行加密。符合所有三個需求。
  • A is wrong: SSE-S3 uses AWS-managed keys — no customer control and no per-use audit trail in KMS. A 錯誤：SSE-S3 使用 AWS 受管金鑰——無客戶控制，KMS 中也無每次使用的稽核追蹤。
  • C is wrong: Client-side encryption adds application complexity and doesn’t provide centralized KMS auditing. C 錯誤：用戶端加密增加應用程式複雜性，且不提供集中式 KMS 稽核。
  • D is wrong: SSE-C requires the customer to manage and transmit keys with every request — high operational burden without centralized audit. D 錯誤：SSE-C 要求客戶在每次請求時管理和傳輸金鑰——高營運負擔且無集中式稽核。
Question 3 — A company’s application runs on EC2 in a private subnet and needs to access a database password stored in AWS Secrets Manager. The security team requires that this traffic never traverse the public internet. What should the solutions architect configure? 問題 3——某公司的應用程式在私有子網路的 EC2 上執行，需要存取儲存在 AWS Secrets Manager 中的資料庫密碼。安全團隊要求此流量絕不得穿越公共網際網路。解決方案架構師應設定什麼？
  1. Assign a public IP to the EC2 instance and use IAM roles for access 為 EC2 執行個體指派公有 IP 並使用 IAM 角色進行存取
  2. Create a NAT Gateway in a public subnet to allow the EC2 instance to reach Secrets Manager 在公有子網路中建立 NAT 閘道，以允許 EC2 執行個體存取 Secrets Manager
  3. Create a VPC Interface Endpoint for Secrets Manager in the private subnet 在私有子網路中為 Secrets Manager 建立 VPC 介面端點
  4. Store the password in S3 and use a VPC Gateway Endpoint to retrieve it 將密碼儲存在 S3 中並使用 VPC 閘道端點擷取
Answer: C 答案：C
  • C is correct: A VPC Interface Endpoint (PrivateLink) for Secrets Manager creates a private ENI in the subnet. Traffic flows entirely within the AWS network, never touching the public internet. C 正確：Secrets Manager 的 VPC 介面端點（PrivateLink）在子網路中建立私有 ENI。流量完全在 AWS 網路內流動，永遠不接觸公共網際網路。
  • A is wrong: A public IP still routes traffic over the internet to reach Secrets Manager’s public endpoint. A 錯誤：公有 IP 仍透過網際網路路由流量以到達 Secrets Manager 的公共端點。
  • B is wrong: NAT Gateway routes outbound traffic to the internet — traffic still leaves the AWS private network. B 錯誤：NAT 閘道將輸出流量路由至網際網路——流量仍離開 AWS 私有網路。
  • D is wrong: S3 Gateway Endpoints only work for S3 and DynamoDB, not Secrets Manager. Also, storing secrets in S3 is an anti-pattern. D 錯誤：S3 閘道端點僅適用於 S3 和 DynamoDB，不適用於 Secrets Manager。而且，將秘密儲存在 S3 中是反模式。
Question 4 — A large enterprise uses AWS Organizations with dozens of accounts. The security team wants to ensure that no account in the organization can disable AWS CloudTrail, regardless of what IAM administrators in those accounts do. What is the most effective solution? 問題 4——某大型企業使用具有數十個帳號的 AWS Organizations。安全團隊希望確保組織中的任何帳號都無法停用 AWS CloudTrail，無論這些帳號中的 IAM 管理員做什麼。最有效的解決方案是什麼？
  1. Create an IAM policy in each account denying cloudtrail:DeleteTrail and attach it to all users 在每個帳號中建立拒絕 cloudtrail:DeleteTrail 的 IAM 政策並附加至所有使用者
  2. Enable AWS Config rules in each account to detect and re-enable CloudTrail if disabled 在每個帳號中啟用 AWS Config 規則，以在 CloudTrail 停用時偵測並重新啟用
  3. Create a Service Control Policy (SCP) denying cloudtrail:DeleteTrail and cloudtrail:StopLogging and attach it to the root OU 建立拒絕 cloudtrail:DeleteTrail 和 cloudtrail:StopLogging 的服務控制政策（SCP）並附加至根組織單位
  4. Use IAM Permission Boundaries on all admin roles to prevent CloudTrail changes 在所有管理員角色上使用 IAM 許可界限以防止 CloudTrail 變更
Answer: C 答案：C
  • C is correct: SCPs attached to the root OU apply to all member accounts and cannot be overridden by any IAM policy within those accounts. No IAM administrator in any member account can override an SCP deny. C 正確：附加至根組織單位的 SCP 套用至所有成員帳號，且無法被這些帳號中的任何 IAM 政策覆寫。任何成員帳號中的 IAM 管理員都無法覆寫 SCP 拒絕。
  • A is wrong: IAM administrators in each account could remove or modify those policies — not tamper-proof. A 錯誤：每個帳號中的 IAM 管理員可以移除或修改這些政策——不防篡改。
  • B is wrong: Config detects drift reactively — there’s a window where CloudTrail is disabled. This is not preventive. B 錯誤：Config 被動式偵測漂移——存在 CloudTrail 停用的時間窗口。這不是預防性的。

Question 5 — A mobile application needs to allow users to sign in with their email and password, and then directly read from a DynamoDB table that contains their personal data. What combination of services should the solutions architect use? 問題 5——行動應用程式需要允許使用者使用電子郵件和密碼登入，然後直接從包含其個人資料的 DynamoDB 資料表讀取。解決方案架構師應使用哪種服務組合？
  1. IAM users for each app user with DynamoDB read permissions 為每個應用程式使用者建立具有 DynamoDB 讀取許可的 IAM 使用者
  2. Cognito User Pool for authentication and Cognito Identity Pool for temporary AWS credentials Cognito 使用者集區用於驗證，Cognito 身分識別集區用於暫時 AWS 憑證
  3. Cognito User Pool only — issue JWT tokens that the app uses to query DynamoDB 僅使用 Cognito 使用者集區——發出應用程式用於查詢 DynamoDB 的 JWT 權杖
  4. IAM Identity Center with a DynamoDB resource policy granting access IAM Identity Center 搭配授予存取權的 DynamoDB 資源政策
Answer: B 答案：B
  • B is correct: Cognito User Pool handles sign-in and issues JWT tokens. The Identity Pool exchanges those JWTs for temporary AWS credentials via STS, scoped to allow DynamoDB reads. This is the standard pattern for mobile apps accessing AWS services directly. B 正確：Cognito 使用者集區處理登入並發出 JWT 權杖。身分識別集區透過 STS 將這些 JWT 兌換為暫時 AWS 憑證，範圍限定為允許 DynamoDB 讀取。這是行動應用程式直接存取 AWS 服務的標準模式。
  • A is wrong: Creating an IAM user per app user is unscalable, insecure, and not how end-user authentication is handled. A 錯誤：為每個應用程式使用者建立 IAM 使用者不具可擴展性、不安全，且不是處理終端使用者驗證的方式。
  • C is wrong: DynamoDB does not accept JWT tokens — it requires AWS SigV4-signed requests. JWTs alone cannot authenticate to DynamoDB. C 錯誤：DynamoDB 不接受 JWT 權杖——它需要 AWS SigV4 簽署的請求。僅憑 JWT 無法向 DynamoDB 驗證。
  • D is wrong: IAM Identity Center is designed for workforce SSO across AWS accounts, not for mobile app end-user authentication. D 錯誤：IAM Identity Center 是為跨 AWS 帳號的員工 SSO 而設計的，不適用於行動應用程式終端使用者驗證。
