1. Domain Overview 領域概覽

Domain 2 tests your ability to select services and configurations that maximize speed, throughput, and scalability. The exam focuses on matching the right performance tool to the workload — choosing the correct EBS type, caching layer, compute family, or network accelerator based on specific requirements like IOPS, latency targets, or data access patterns.

領域 2 測試您選擇服務和設定以最大化速度、輸送量和擴展性的能力。考試重點在於為工作負載配對正確的效能工具——根據 IOPS、延遲目標或資料存取模式等特定需求，選擇正確的 EBS 類型、快取層、運算系列或網路加速器。

2. Key AWS Services 主要 AWS 服務

Compute 運算

EC2 Instance Families EC2 執行個體系列

  • C-series (c6i, c7g): Compute-optimized; CPU-bound workloads, batch processing, web servers C 系列（c6i, c7g）：運算最佳化；CPU 密集工作負載、批次處理、Web 伺服器
  • R/X-series (r6i, x2idn): Memory-optimized; in-memory databases, real-time big data, SAP HANA R/X 系列（r6i, x2idn）：記憶體最佳化；記憶體內資料庫、即時大數據、SAP HANA
  • I/D-series (i4i, d3): Storage-optimized; high sequential I/O, data warehousing, Hadoop I/D 系列（i4i, d3）：儲存最佳化；高循序 I/O、資料倉儲、Hadoop
  • P/G/Inf-series: Accelerated computing; ML training (P), GPU graphics (G), ML inference (Inf) P/G/Inf 系列：加速運算；ML 訓練（P）、GPU 圖形（G）、ML 推論（Inf）
  • Exam rule: match instance family to bottleneck — CPU-bound → C; memory-bound → R; IOPS-bound → I; ML → P/Inf 考試規則：將執行個體系列對應至瓶頸——CPU 密集 → C；記憶體密集 → R；IOPS 密集 → I；ML → P/Inf

Enhanced Networking 增強型網路

  • ENA (Elastic Network Adapter): up to 100 Gbps, lower latency, lower jitter — available on most modern instance types ENA（彈性網路轉接器）：最高 100 Gbps、較低延遲、較低抖動——適用於大多數現代執行個體類型
  • EFA (Elastic Fabric Adapter): HPC and ML workloads requiring tightly coupled node communication (MPI); only on supported instances EFA（彈性網路結構轉接器）：需要緊密耦合節點通訊（MPI）的 HPC 和 ML 工作負載；僅限支援的執行個體
  • Exam rule: “HPC cluster with inter-node communication” → EFA; general high bandwidth → ENA 考試規則：「具有節點間通訊的 HPC 叢集」→ EFA；一般高頻寬 → ENA

Lambda

  • Memory allocation (128 MB – 10,240 MB) also scales CPU proportionally 記憶體配置（128 MB – 10,240 MB）也按比例擴展 CPU
  • Provisioned Concurrency: pre-warms execution environments — eliminates cold starts for latency-sensitive workloads 佈建並行：預先暖機執行環境——消除對延遲敏感工作負載的冷啟動
  • Max timeout: 15 minutes  |  Max memory: 10,240 MB 最長逾時：15 分鐘 | 最大記憶體：10,240 MB
  • Default concurrency limit: 1,000 per region (soft limit, can be raised) 預設並行限制：每個區域 1,000（軟性限制，可提高）
  • Exam trap: “Lambda takes too long on first request” → Provisioned Concurrency, not increasing memory 考試陷阱：「Lambda 在第一個請求時花費太長時間」→ 佈建並行，而非增加記憶體

ECS / EKS / Fargate

  • ECS: AWS-native container orchestration ECS：AWS 原生容器協調
  • EKS: Kubernetes; use when team already uses K8s or needs K8s ecosystem EKS：Kubernetes；當團隊已使用 K8s 或需要 K8s 生態系統時使用
  • Fargate: serverless compute for containers (no EC2 management) Fargate：容器的無伺服器運算（無需管理 EC2）
  • Exam rule: “no server management for containers” → Fargate; “K8s required” → EKS; “AWS-native containers” → ECS 考試規則：「容器無需伺服器管理」→ Fargate；「需要 K8s」→ EKS；「AWS 原生容器」→ ECS

S3 Performance S3 效能

  • Default: 3,500 PUT/COPY/POST/DELETE and 5,500 GET/HEAD per second per prefix 預設：每個字首每秒 3,500 PUT/COPY/POST/DELETE 和 5,500 GET/HEAD
  • Add prefixes to scale horizontally (distribute objects across partitions) 新增字首以水平擴展（跨分割區分配物件）
  • Multipart Upload: recommended for >100 MB, required for >5 GB 多部分上傳：建議用於 >100 MB，>5 GB 時必須使用
  • S3 Transfer Acceleration: routes uploads through CloudFront edge locations — use for long-distance uploads S3 傳輸加速：透過 CloudFront 邊緣位置路由上傳——用於長距離上傳
  • S3 Byte-Range Fetches: parallel downloads of object parts — improves GET throughput S3 位元組範圍擷取：平行下載物件部分——提升 GET 輸送量

EFS (Elastic File System) EFS（彈性檔案系統）

  • Bursting: throughput scales with storage size 爆量：輸送量隨儲存大小擴展
  • Provisioned: set throughput independently of storage — use when workload needs more than storage tier provides 佈建：獨立於儲存設定輸送量——當工作負載需求超過儲存層提供時使用
  • Elastic: auto-scales throughput up/down; best for unpredictable workloads (recommended default) 彈性：自動向上/向下擴展輸送量；最適合無法預測的工作負載（建議預設）
  • Max I/O mode: higher aggregate throughput, higher latency — only for massively parallel workloads (10,000+ clients) 最大 I/O 模式：更高的彙總輸送量、更高延遲——僅用於大量平行工作負載（10,000 個以上用戶端）
  • Exam rule: “NFS shared file system for Linux” → EFS; “unpredictable throughput” → Elastic mode 考試規則：「Linux 的 NFS 共用檔案系統」→ EFS；「無法預測的輸送量」→ 彈性模式

FSx for Windows File Server FSx for Windows 檔案伺服器

  • SMB protocol, NTFS, Active Directory integration SMB 通訊協定、NTFS、Active Directory 整合
  • Exam rule: “Windows workloads needing shared file storage” or “SMB protocol” → FSx for Windows 考試規則：「需要共用檔案儲存的 Windows 工作負載」或「SMB 通訊協定」→ FSx for Windows

FSx for Lustre

  • High-performance parallel file system; sub-millisecond latency; hundreds of GB/s throughput 高效能平行檔案系統；毫秒以下延遲；數百 GB/s 輸送量
  • Integrates natively with S3 — read/write S3 as a filesystem 與 S3 原生整合——以檔案系統方式讀寫 S3
  • Exam rule: “HPC”, “ML training with large datasets”, “high-throughput compute” → FSx for Lustre 考試規則：「HPC」、「使用大型資料集的 ML 訓練」、「高輸送量運算」→ FSx for Lustre

DynamoDB Accelerator (DAX) DynamoDB 加速器

  • In-memory cache purpose-built for DynamoDB; microsecond read latency 專為 DynamoDB 建置的記憶體快取；微秒讀取延遲
  • Write-through cache; API-compatible with DynamoDB (minimal code change) 寫入穿透快取；與 DynamoDB API 相容（最少程式碼變更）
  • Does NOT help with write-heavy workloads or strongly consistent reads 對寫入密集工作負載或強一致性讀取無幫助
  • Exam rule: “DynamoDB reads are too slow” or “microsecond latency for DynamoDB” → DAX 考試規則：「DynamoDB 讀取太慢」或「DynamoDB 需要微秒延遲」→ DAX

RDS Read Replicas RDS 唯讀複本

  • Asynchronous replication; offloads read traffic from primary 非同步複寫；從主要執行個體分擔讀取流量
  • Exam rule: “read-heavy workload causing performance issues on RDS” → add read replicas 考試規則：「讀取密集工作負載導致 RDS 效能問題」→ 新增唯讀複本

Networking & Content Delivery 網路與內容交付

CloudFront

  • Global CDN with 450+ edge locations; caches static and dynamic content 擁有 450 個以上邊緣位置的全球 CDN；快取靜態和動態內容
  • Origins: S3, ALB, EC2, API Gateway, custom HTTP 來源：S3、ALB、EC2、API Gateway、自訂 HTTP
  • Origin Shield: extra caching layer between edges and origin — reduces origin load 來源防護：邊緣與來源之間的額外快取層——減少來源負載
  • Exam rule: “reduce latency for global users for HTTP/HTTPS content” → CloudFront 考試規則：「降低全球使用者的 HTTP/HTTPS 內容延遲」→ CloudFront

Global Accelerator 全球加速器

  • Routes traffic over AWS backbone to optimal endpoints using anycast IP 使用任播 IP 透過 AWS 骨幹網路將流量路由至最佳端點
  • Provides 2 static global IPs — no DNS propagation delay on failover 提供 2 個靜態全球 IP——容錯移轉時無 DNS 傳播延遲
  • Works for TCP/UDP (not HTTP-specific like CloudFront) 適用於 TCP/UDP（不像 CloudFront 那樣特定於 HTTP）
  • Exam rule: “static IP for global routing” or “non-HTTP protocol” → Global Accelerator; “HTTP content caching” → CloudFront 考試規則：「全球路由的靜態 IP」或「非 HTTP 通訊協定」→ 全球加速器；「HTTP 內容快取」→ CloudFront

VPC Endpoints VPC 端點

  • Interface Endpoint (PrivateLink): ENI in subnet; supports most AWS services; costs money per hour 介面端點（PrivateLink）：子網路中的 ENI；支援大多數 AWS 服務；按小時收費
  • Gateway Endpoint: adds route to route table; only S3 and DynamoDB; free 閘道端點：在路由表中新增路由；僅限 S3 和 DynamoDB；免費
  • Exam rule: “EC2 accessing S3 without internet” → Gateway Endpoint (free); anything else → Interface Endpoint 考試規則：「EC2 不經網際網路存取 S3」→ 閘道端點（免費）；其他情況 → 介面端點

Direct Connect 專線連線

  • Dedicated private network connection from on-premises to AWS 從本地端到 AWS 的專用私有網路連線
  • Consistent latency, higher bandwidth, lower per-GB cost than internet VPN 一致延遲、更高頻寬、比網際網路 VPN 更低的每 GB 費用
  • Exam rule: “consistent network performance to AWS” or “high-throughput hybrid connectivity” → Direct Connect 考試規則：「對 AWS 一致的網路效能」或「高輸送量混合連線」→ Direct Connect

Streaming & Messaging 串流與訊息傳遞

Kinesis Data Streams Kinesis 資料串流

  • Real-time streaming; ordered within a shard; retention up to 365 days 即時串流；在分片內有序；保留期最長 365 天
  • Each shard = 1 MB/s write, 2 MB/s read 每個分片 = 1 MB/s 寫入、2 MB/s 讀取
  • Enhanced Fan-Out: dedicated 2 MB/s per consumer per shard — eliminates consumer competition 增強型展開：每個消費者每個分片專用 2 MB/s——消除消費者競爭

Kinesis Data Firehose Kinesis Data Firehose

  • Managed delivery to S3, Redshift, OpenSearch, Splunk; no shards to manage 受管交付至 S3、Redshift、OpenSearch、Splunk；無需管理分片
  • Near-real-time (60-second buffer minimum) 接近即時（最少 60 秒緩衝）
  • Exam rule: “stream data to S3/Redshift without managing infrastructure” → Firehose; “real-time processing with custom consumers” → Data Streams 考試規則：「無需管理基礎設施即可將資料串流至 S3/Redshift」→ Firehose；「使用自訂消費者的即時處理」→ Data Streams

3. Architecture Patterns 架構模式

Pattern 1: Read-Heavy Application with Caching 模式 1：使用快取的讀取密集應用程式

Services: Route 53 → ALB → EC2 ASG → ElastiCache Redis → RDS

服務：Route 53 → ALB → EC2 ASG → ElastiCache Redis → RDS

Why it works: Read requests hit ElastiCache first (sub-millisecond). Only cache misses hit RDS. Dramatically reduces DB load and response time.

為何有效：讀取請求先命中 ElastiCache（毫秒以下）。僅快取未命中才存取 RDS。大幅降低資料庫負載和回應時間。

Exam signal: “database is a bottleneck”, “reduce read latency”, “high read-to-write ratio” → add ElastiCache in front of RDS.

考試信號：「資料庫是瓶頸」、「降低讀取延遲」、「高讀寫比例」→ 在 RDS 前新增 ElastiCache。

Pattern 2: High-Throughput Data Ingestion Pipeline 模式 2：高輸送量資料擷取管道

Services: Producers → Kinesis Data Streams → Lambda / EC2 consumers → S3 / DynamoDB / Redshift

服務：生產者 → Kinesis Data Streams → Lambda/EC2 消費者 → S3/DynamoDB/Redshift

Why it works: Kinesis buffers high-volume real-time data; multiple consumers independently process the same stream; replay is possible.

為何有效：Kinesis 緩衝大量即時資料；多個消費者獨立處理同一串流；可重播。

Exam signal: “ingest millions of events per second”, “multiple downstream systems need the same data”, “real-time processing” → Kinesis Data Streams.

考試信號：「每秒擷取數百萬個事件」、「多個下游系統需要相同資料」、「即時處理」→ Kinesis Data Streams。

Pattern 3: Global API Acceleration 模式 3：全球 API 加速

Services: Route 53 → Global Accelerator → ALB in multiple regions → EC2/Lambda

服務：Route 53 → 全球加速器 → 多個區域的 ALB → EC2/Lambda

Why it works: Traffic enters AWS network at nearest edge, then travels the optimized AWS backbone. Reduces latency vs. public internet routing.

為何有效：流量在最近的邊緣進入 AWS 網路，然後透過最佳化的 AWS 骨幹傳輸。相較於公共網際網路路由降低延遲。

Exam signal: “global users experiencing high latency”, “non-HTTP application”, “need static IP for global routing” → Global Accelerator.

考試信號：「全球使用者遭遇高延遲」、「非 HTTP 應用程式」、「需要全球路由的靜態 IP」→ 全球加速器。

Pattern 4: HPC / ML Training Architecture 模式 4：HPC / ML 訓練架構

Services: EC2 P/G/Inf instances + EFA → FSx for Lustre (linked to S3) → S3

服務：EC2 P/G/Inf 執行個體 + EFA → FSx for Lustre（連結至 S3）→ S3

Why it works: EFA enables low-latency inter-node communication. FSx for Lustre provides high-throughput shared storage. S3 feeds training data into FSx.

為何有效：EFA 啟用低延遲節點間通訊。FSx for Lustre 提供高輸送量共用儲存。S3 將訓練資料饋入 FSx。

Exam signal: “ML training”, “HPC cluster”, “hundreds of GB/s storage throughput” → FSx for Lustre + EFA-enabled instances.

考試信號：「ML 訓練」、「HPC 叢集」、「數百 GB/s 儲存輸送量」→ FSx for Lustre + 啟用 EFA 的執行個體。

Pattern 5: Serverless High-Performance API 模式 5：無伺服器高效能 API

Services: API Gateway → Lambda (Provisioned Concurrency) → DynamoDB + DAX

服務：API Gateway → Lambda（佈建並行）→ DynamoDB + DAX

Why it works: Provisioned Concurrency eliminates cold starts. DAX provides microsecond DynamoDB reads. No servers to scale.

為何有效：佈建並行消除冷啟動。DAX 提供微秒 DynamoDB 讀取。無需擴展伺服器。

Exam signal: “serverless API with consistent sub-10ms response time” → Lambda + Provisioned Concurrency + DAX.

考試信號：「具一致性 10 毫秒以下回應時間的無伺服器 API」→ Lambda + 佈建並行 + DAX。

4. Service Comparisons 服務比較

CloudFront vs. Global Accelerator

  • CloudFront = HTTP/HTTPS CDN, caches content at edge, reduces origin load CloudFront = HTTP/HTTPS CDN，在邊緣快取內容，減少來源負載
  • Global Accelerator = TCP/UDP routing over AWS backbone, no caching, static IPs 全球加速器 = 透過 AWS 骨幹的 TCP/UDP 路由，無快取，靜態 IP
  • Exam rule: “cache assets globally” or “HTTP API latency” → CloudFront; “static IPs” or “non-HTTP” → Global Accelerator 考試規則：「全球快取資產」或「HTTP API 延遲」→ CloudFront；「靜態 IP」或「非 HTTP」→ 全球加速器

ElastiCache Redis vs. Memcached

  • Redis = persistence, replication, complex data types, Multi-AZ, pub/sub Redis = 持久性、複寫、複雜資料類型、多可用區域、Pub/Sub
  • Memcached = simple key-value, multi-threaded, pure horizontal scaling Memcached = 簡單鍵值、多執行緒、純水平擴展
  • Exam rule: “sorted leaderboard”, “session persistence”, “pub/sub”, “Multi-AZ cache” → Redis; “simple scalable cache” → Memcached 考試規則：「排序排行榜」、「工作階段持久性」、「Pub/Sub」、「多可用區域快取」→ Redis；「簡單可擴展快取」→ Memcached

gp3 vs. io2 EBS

  • gp3 = 3,000–16,000 IOPS, cost-effective general purpose gp3 = 3,000–16,000 IOPS，符合成本效益的一般用途
  • io2 = up to 64,000 IOPS, consistent sub-ms latency, 99.999% durability io2 = 最高 64,000 IOPS、一致的毫秒以下延遲、99.999% 持久性
  • Exam rule: >16,000 IOPS or “mission-critical consistent latency” → io2; everything else → gp3 考試規則：>16,000 IOPS 或「關鍵任務一致延遲」→ io2；其他情況 → gp3

Kinesis Data Streams vs. Kinesis Firehose Kinesis Data Streams vs. Kinesis Firehose

  • Data Streams = real-time, custom consumers, ordered per shard, replay, code required Data Streams = 即時、自訂消費者、每個分片有序、可重播、需要程式碼
  • Firehose = near-real-time, managed delivery to destinations, no code needed Firehose = 接近即時、受管交付至目的地、無需程式碼
  • Exam rule: “real-time processing with custom logic” → Data Streams; “load to S3/Redshift automatically” → Firehose 考試規則：「使用自訂邏輯的即時處理」→ Data Streams；「自動載入至 S3/Redshift」→ Firehose

EFS vs. FSx for Lustre

  • EFS = general-purpose Linux shared file system, multi-AZ, POSIX-compliant EFS = 一般用途 Linux 共用檔案系統、多可用區域、POSIX 相容
  • FSx for Lustre = HPC-grade throughput, S3 integration, sub-millisecond latency FSx for Lustre = HPC 等級輸送量、S3 整合、毫秒以下延遲
  • Exam rule: “shared file storage for general Linux workloads” → EFS; “HPC or ML needing maximum throughput” → FSx for Lustre 考試規則：「一般 Linux 工作負載的共用檔案儲存」→ EFS；「需要最大輸送量的 HPC 或 ML」→ FSx for Lustre

DAX vs. ElastiCache for DynamoDB

  • DAX = purpose-built for DynamoDB, drop-in replacement, API-compatible, read-through/write-through DAX = 專為 DynamoDB 建置、即插即用替換、API 相容、讀取穿透/寫入穿透
  • ElastiCache = general-purpose cache, requires application-level cache logic ElastiCache = 一般用途快取，需要應用程式層級的快取邏輯
  • Exam rule: “speed up DynamoDB reads with minimal code change” → DAX; “cache data from multiple sources” → ElastiCache 考試規則：「以最少程式碼變更加速 DynamoDB 讀取」→ DAX；「快取來自多個來源的資料」→ ElastiCache

5. Exam Traps & Gotchas 考試陷阱與注意事項

  1. Lambda cold starts vs. memory. Increasing memory speeds up execution but does NOT eliminate cold starts. Cold starts require Provisioned Concurrency. Lambda 冷啟動 vs. 記憶體。增加記憶體可加快執行速度，但不能消除冷啟動。消除冷啟動需要佈建並行。
  2. gp2 IOPS tied to volume size; gp3 is not. A 100 GB gp2 gets only 300 IOPS baseline. A 100 GB gp3 gets 3,000 IOPS. Small volumes needing high IOPS → gp3 or io2. gp2 的 IOPS 與磁碟區大小掛鉤；gp3 則不然。100 GB gp2 僅有 300 IOPS 基準。100 GB gp3 有 3,000 IOPS。需要高 IOPS 的小型磁碟區 → gp3 或 io2。
  3. DAX does not help write performance. DAX caches reads. Write-heavy DynamoDB workloads don’t benefit from DAX. DAX 對寫入效能無幫助。DAX 快取讀取。寫入密集的 DynamoDB 工作負載無法從 DAX 受益。
  4. S3 performance is per prefix, not per bucket. All objects under one prefix = limited to 5,500 GET/s. Distribute across multiple prefixes to multiply throughput. S3 效能按字首計算，而非按儲存貯體計算。同一字首下的所有物件 = 限制為 5,500 GET/秒。跨多個字首分佈以倍增輸送量。
  5. Kinesis Enhanced Fan-Out vs. standard GetRecords. Standard polling shares 2 MB/s per shard across all consumers. Enhanced Fan-Out gives each consumer dedicated 2 MB/s. “Multiple consumers with high lag” → Enable Enhanced Fan-Out. Kinesis 增強型展開 vs. 標準 GetRecords。標準輪詢跨所有消費者共享每分片 2 MB/s。增強型展開給每個消費者專用 2 MB/s。「多個消費者延遲高」→ 啟用增強型展開。
  6. Global Accelerator does not cache. It only routes traffic faster. If the question mentions caching or content delivery, CloudFront is the answer. 全球加速器不快取。它只是更快地路由流量。若問題提到快取或內容交付，CloudFront 才是答案。
  7. EFS Max I/O has higher latency. Not a performance upgrade for all workloads. Only for massively parallel workloads (10,000+ clients). EFS 最大 I/O 有更高延遲。並非所有工作負載的效能升級。僅適用於大量平行工作負載（10,000 個以上用戶端）。
  8. st1 and sc1 cannot be boot volumes. Only gp2, gp3, io1, io2 can be root/boot volumes. st1 和 sc1 不能作為開機磁碟區。只有 gp2、gp3、io1、io2 可以作為根/開機磁碟區。

7. Practice Questions 練習題

Question 1 — A company runs a read-heavy web application backed by an RDS MySQL database. The database is experiencing high CPU utilization due to a large number of repetitive read queries. The application team cannot change the application code significantly. What should a solutions architect recommend? 問題 1——某公司執行以 RDS MySQL 資料庫為後端的讀取密集 Web 應用程式。由於大量重複讀取查詢，資料庫的 CPU 使用率很高。應用程式團隊無法大幅變更應用程式程式碼。解決方案架構師應建議什麼？
  1. Add RDS Read Replicas and update the application to route reads to the replica endpoint 新增 RDS 唯讀複本並更新應用程式將讀取路由至複本端點
  2. Enable RDS Multi-AZ to distribute read traffic to the standby instance 啟用 RDS 多可用區域將讀取流量分配至備用執行個體
  3. Deploy an ElastiCache Memcached cluster in front of RDS 在 RDS 前部署 ElastiCache Memcached 叢集
  4. Deploy an ElastiCache Redis cluster in front of RDS and use lazy loading 在 RDS 前部署 ElastiCache Redis 叢集並使用延遲載入
Answer: D 答案：D
  • D is correct: ElastiCache Redis with lazy loading caches frequent read results, dramatically reducing RDS CPU. Redis is preferred for its replication and persistence. D 正確：ElastiCache Redis 搭配延遲載入快取頻繁讀取結果，大幅降低 RDS CPU。Redis 因其複寫和持久性而更受青睞。
  • A is wrong: Read Replicas still hit the database for every query — repetitive queries are not cached. A 錯誤：唯讀複本對每個查詢仍存取資料庫——重複查詢不會被快取。
  • B is wrong: Multi-AZ standby is not readable. It exists only for failover. B 錯誤：多可用區域備用執行個體無法讀取。它僅用於容錯移轉。
  • C is wrong: Memcached could work but lacks replication and persistence. Redis is the better default for production caching. C 錯誤：Memcached 可以運作，但缺乏複寫和持久性。Redis 是生產快取的更好預設選擇。
Question 2 — A machine learning team needs to train a model on a 50 TB dataset stored in S3. Training jobs run on a cluster of EC2 P4d instances. The training requires extremely high throughput I/O and low latency access to the dataset. What storage solution should a solutions architect recommend? 問題 2——機器學習團隊需要在儲存於 S3 的 50 TB 資料集上訓練模型。訓練工作在 EC2 P4d 執行個體叢集上執行。訓練需要極高輸送量 I/O 和對資料集的低延遲存取。解決方案架構師應建議什麼儲存方案？
  1. Mount the S3 bucket directly using S3 File Gateway on each instance 在每個執行個體上使用 S3 File Gateway 直接掛載 S3 儲存貯體
  2. Copy the dataset to EBS gp3 volumes attached to each instance 將資料集複製到連接至每個執行個體的 EBS gp3 磁碟區
  3. Use Amazon FSx for Lustre linked to the S3 bucket 使用連結至 S3 儲存貯體的 Amazon FSx for Lustre
  4. Use Amazon EFS with Max I/O performance mode 使用最大 I/O 效能模式的 Amazon EFS
Answer: C 答案：C
  • C is correct: FSx for Lustre provides sub-millisecond latency and hundreds of GB/s throughput with native S3 integration. Designed exactly for HPC/ML workloads. C 正確：FSx for Lustre 提供毫秒以下延遲和數百 GB/s 輸送量，並具有原生 S3 整合。專為 HPC/ML 工作負載設計。
  • A is wrong: S3 File Gateway adds latency and is meant for hybrid on-premises scenarios. A 錯誤：S3 File Gateway 增加延遲，適用於混合本地端場景。
  • B is wrong: EBS volumes are per-instance. Copying 50 TB to each instance is impractical and doesn’t support shared access. B 錯誤：EBS 磁碟區是每個執行個體專屬的。將 50 TB 複製到每個執行個體不切實際，且不支援共用存取。
  • D is wrong: EFS Max I/O has higher latency than FSx for Lustre and lower aggregate throughput. Not designed for this scale. D 錯誤：EFS 最大 I/O 的延遲高於 FSx for Lustre，彙總輸送量也較低。不是為此規模而設計的。
Question 3 — A company has a DynamoDB table that serves a leaderboard for a gaming application. As the game grew, read latency has increased to 5–8 ms, causing visible lag in the UI. The team needs microsecond read latency. What should the solutions architect recommend? 問題 3——某公司有一個 DynamoDB 資料表，為遊戲應用程式提供排行榜。隨著遊戲成長，讀取延遲已增加至 5–8 毫秒，導致 UI 出現明顯延遲。團隊需要微秒讀取延遲。解決方案架構師應建議什麼？
  1. Add a Global Secondary Index to the DynamoDB table 在 DynamoDB 資料表中新增全域次要索引
  2. Switch to DynamoDB on-demand capacity mode 切換至 DynamoDB 隨需容量模式
  3. Deploy Amazon DynamoDB Accelerator (DAX) in front of the table 在資料表前部署 Amazon DynamoDB Accelerator（DAX）
  4. Add an ElastiCache Redis cluster and implement a cache-aside pattern 新增 ElastiCache Redis 叢集並實作旁路快取模式
Answer: C 答案：C
  • C is correct: DAX is purpose-built for DynamoDB and provides microsecond read latency. API-compatible with DynamoDB, so minimal code changes are required. C 正確：DAX 專為 DynamoDB 建置，提供微秒讀取延遲。與 DynamoDB API 相容，因此只需最少程式碼變更。
  • A is wrong: A GSI improves query flexibility, not read latency. It doesn’t cache anything. A 錯誤：全域次要索引提升查詢彈性，而非讀取延遲。它不快取任何內容。
  • B is wrong: On-demand mode adjusts capacity, not latency. The bottleneck is latency, not throughput. B 錯誤：隨需模式調整容量，而非延遲。瓶頸是延遲，而非輸送量。
  • D is wrong: ElastiCache Redis requires significant application code changes. DAX is the simpler, purpose-built solution. D 錯誤：ElastiCache Redis 需要大量應用程式程式碼變更。DAX 是更簡單的專用解決方案。
Question 4 — A media company streams video content to users worldwide. Videos are stored in S3 in us-east-1. Users in Asia and Europe are experiencing long load times. The company wants to reduce latency with minimal code changes. What should the solutions architect recommend? 問題 4——某媒體公司向全球使用者串流視訊內容。影片儲存在 us-east-1 的 S3 中。亞洲和歐洲的使用者遭遇較長的載入時間。公司希望以最少程式碼變更降低延遲。解決方案架構師應建議什麼？
  1. Enable S3 Transfer Acceleration on the bucket 在儲存貯體上啟用 S3 傳輸加速
  2. Use Amazon CloudFront with an S3 origin 使用以 S3 為來源的 Amazon CloudFront
  3. Use AWS Global Accelerator pointing to the S3 bucket 使用指向 S3 儲存貯體的 AWS 全球加速器
  4. Replicate the S3 bucket to regions in Asia and Europe using S3 CRR 使用 S3 CRR 將 S3 儲存貯體複寫至亞洲和歐洲的區域
Answer: B 答案：B
  • B is correct: CloudFront caches video content at 450+ edge locations globally. Users are served from the nearest edge, dramatically reducing load times. B 正確：CloudFront 在全球 450 個以上邊緣位置快取視訊內容。使用者從最近的邊緣提供服務，大幅降低載入時間。
  • A is wrong: S3 Transfer Acceleration speeds up uploads to S3, not downloads to end users. A 錯誤：S3 傳輸加速加快上傳至 S3 的速度，而非向終端使用者的下載速度。
  • C is wrong: Global Accelerator doesn’t cache content. The origin still serves every request. C 錯誤：全球加速器不快取內容。來源仍然服務每個請求。
  • D is wrong: CRR adds operational complexity and cost. CloudFront achieves the same goal with a single bucket. D 錯誤：CRR 增加營運複雜性和成本。CloudFront 以單一儲存貯體達到相同目標。
Question 5 — A company processes clickstream events from its website. Multiple downstream systems need to process the same events independently — a real-time dashboard, a fraud detection system, and a data archiving pipeline. Events must be processed in order and replay must be possible. What should a solutions architect recommend? 問題 5——某公司處理來自其網站的點擊串流事件。多個下游系統需要獨立處理相同事件——即時儀表板、詐欺偵測系統和資料封存管道。事件必須按順序處理，且必須可以重播。解決方案架構師應建議什麼？
  1. Use Amazon SQS Standard queues with one queue per consumer 使用每個消費者一個佇列的 Amazon SQS 標準佇列
  2. Use Amazon SNS with three SQS queues as subscribers 使用 Amazon SNS 搭配三個 SQS 佇列作為訂閱者
  3. Use Amazon Kinesis Data Streams with Enhanced Fan-Out enabled 使用啟用增強型展開的 Amazon Kinesis Data Streams
  4. Use Amazon EventBridge with three separate rules and targets 使用具有三個獨立規則和目標的 Amazon EventBridge
Answer: C 答案：C
  • C is correct: Kinesis Data Streams preserves order within a shard, supports replay, and Enhanced Fan-Out gives each consumer dedicated 2 MB/s per shard. Perfect for multi-consumer ordered replayable streaming. C 正確：Kinesis Data Streams 在分片內保留順序、支援重播，增強型展開給每個消費者每個分片專用 2 MB/s。完美適用於多消費者有序可重播串流。
  • A is wrong: A message consumed from one SQS queue is not available to others. Duplicating events to multiple queues is an anti-pattern. A 錯誤：從一個 SQS 佇列消費的訊息無法供其他人使用。將事件複製到多個佇列是反模式。
  • B is wrong: SNS + SQS fanout distributes events but SQS doesn’t guarantee order and doesn’t support replay. B 錯誤：SNS + SQS 展開分配事件，但 SQS 不保證順序且不支援重播。
  • D is wrong: EventBridge is excellent for event routing but does not support ordered processing or stream replay. D 錯誤：EventBridge 非常適合事件路由，但不支援有序處理或串流重播。
