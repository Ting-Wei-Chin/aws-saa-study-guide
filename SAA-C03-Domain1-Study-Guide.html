SAA-C03 — Domain 1: Design Resilient Architectures 領域 1：設計具彈性的架構

AWS Certified Solutions Architect – Associate  |  Deep-Dive Study Guide  |  26% of Exam AWS 認證解決方案架構師 – 助理  |  深度學習指南  |  占考試 26%

1. Domain Overview 領域概覽

Domain 1 tests your ability to select AWS services and configurations that keep applications running through failures, traffic spikes, and disasters. The exam focuses on which combination of services achieves a specific resilience requirement — not just that HA exists, but knowing when Multi-AZ vs. Multi-Region vs. Auto Scaling is the right tool, and at what cost.

領域 1 測試您選擇 AWS 服務和設定的能力，以確保應用程式在故障、流量激增和災難情況下持續運作。考試重點在於哪種服務組合能達到特定的彈性需求——不只是知道高可用性的存在，而是要了解何時應使用多可用區域、多區域或 Auto Scaling，以及各自的成本考量。

2. Key AWS Services 主要 AWS 服務

Compute 運算

EC2

  • Backbone of most resilience architectures on the exam 考試中大多數彈性架構的核心服務
  • Key exam configs: placement groups (Cluster vs. Spread vs. Partition), instance store vs. EBS, and Spot interruption handling 關鍵考試設定：置放群組（叢集 vs. 分散 vs. 分割）、執行個體存放區 vs. EBS，以及 Spot 中斷處理
  • Spread placement group = max 7 instances per AZ per group — exam loves this limit 分散置放群組 = 每個群組每個可用區域最多 7 個執行個體——考試常考此限制

Auto Scaling Groups (ASG) Auto Scaling 群組

  • Automatically replaces unhealthy instances; scales in/out based on policies 自動取代狀況不佳的執行個體；根據政策水平擴展或縮減
  • Use with ELB health checks (not just EC2 health checks) to replace instances that fail at the app layer, not just the OS layer 搭配 ELB 運作狀態檢查（而非僅 EC2 運作狀態檢查），以取代應用程式層故障的執行個體，而非僅限於作業系統層
  • Key number: default cooldown period = 300 seconds 關鍵數字：預設冷卻期間 = 300 秒

EC2 Instance Store EC2 執行個體存放區

  • Ephemeral — data lost on stop/terminate/hardware failure 暫時性存放區——執行個體停止、終止或硬體故障時資料將遺失
  • Exam trap: high IOPS workload → instance store looks right, but if durability is required, it’s wrong 考試陷阱：高 IOPS 工作負載 → 執行個體存放區看似正確，但若需要持久性則為錯誤答案

Load Balancing 負載平衡

Application Load Balancer (ALB) 應用程式負載平衡器

  • Layer 7; routes by path, host, headers, query strings 第 7 層；依路徑、主機、標頭、查詢字串路由
  • Use for HTTP/HTTPS microservices, containerized apps, WebSocket 適用於 HTTP/HTTPS 微服務、容器化應用程式、WebSocket

Network Load Balancer (NLB) 網路負載平衡器

  • Layer 4; ultra-low latency, handles millions of requests/second 第 4 層；超低延遲，可處理每秒數百萬個請求
  • Use when: static IP required, TCP/UDP traffic, extreme performance needed 適用場景：需要靜態 IP、TCP/UDP 流量、需要極高效能
  • Only ELB type that supports Elastic IP assignment 唯一支援彈性 IP 指派的 ELB 類型

Gateway Load Balancer (GWLB) 閘道負載平衡器

  • Used specifically for inline network appliances (firewalls, IDS/IPS) 專門用於串聯式網路設備（防火牆、入侵偵測/防護系統）
  • Exam: “third-party virtual appliance” → GWLB, not ALB/NLB 考試：「第三方虛擬設備」→ GWLB，非 ALB/NLB

Classic Load Balancer 傳統負載平衡器

  • Legacy — only appears on exam when the answer is “migrate away from it” 舊版——考試中只有在答案是「遷移離開」時才會出現

Database Resilience 資料庫彈性

RDS Multi-AZ RDS 多可用區域

  • Synchronous replication to standby in another AZ 同步複寫至另一個可用區域的備用執行個體
  • Standby is NOT readable — it’s only for failover 備用執行個體無法讀取——僅用於容錯移轉
  • Failover takes 1–2 minutes; DNS endpoint automatically redirects 容錯移轉需要 1–2 分鐘；DNS 端點自動重新導向
  • Exam trap: “read performance” → Multi-AZ is wrong, Read Replica is right 考試陷阱：「讀取效能」→ 多可用區域為錯誤，唯讀複本才正確

RDS Read Replicas RDS 唯讀複本

  • Asynchronous replication; replicas ARE readable 非同步複寫；複本可供讀取
  • Can be promoted to standalone DB (used in DR scenarios) 可升級為獨立資料庫（用於災難復原場景）
  • Can be in a different region (cross-region read replica) 可位於不同區域（跨區域唯讀複本）

Aurora

  • 6 copies of data across 3 AZs automatically 自動在 3 個可用區域儲存 6 份資料副本
  • Up to 15 read replicas; failover to replica in ~30 seconds (vs. RDS ~60–120s) 最多 15 個唯讀複本；容錯移轉至複本約需 30 秒（相較 RDS 約 60–120 秒）
  • Aurora Global Database: replication to secondary region in <1 second; RPO ~1s, RTO <1 min Aurora 全球資料庫：複寫至次要區域時間 <1 秒；RPO 約 1 秒，RTO <1 分鐘
  • Exam: “fastest RDS failover” → Aurora; “global low-latency reads” → Aurora Global 考試：「最快 RDS 容錯移轉」→ Aurora；「全球低延遲讀取」→ Aurora 全球資料庫

DynamoDB

  • Multi-AZ by default (within a region) 預設多可用區域（在單一區域內）
  • Global Tables: multi-region, multi-active replication (active-active, not active-passive) 全球資料表：多區域、多主動複寫（主動-主動，非主動-被動）
  • On-demand mode = resilience against unpredictable traffic spikes without capacity planning 隨需模式 = 無需容量規劃，即可因應無法預測的流量激增

Storage 儲存

S3

  • 11 nines durability, replicated across minimum 3 AZs (except S3 One Zone-IA) 11 個 9 的持久性，至少跨 3 個可用區域複寫（S3 單一可用區域-IA 除外）
  • Versioning + MFA Delete = protection against accidental deletion 版本控制 + MFA 刪除 = 防止意外刪除的保護措施
  • Cross-Region Replication (CRR): requires versioning enabled on both buckets; does NOT retroactively replicate existing objects 跨區域複寫（CRR）：需在兩個儲存貯體啟用版本控制；不會追溯複寫現有物件
  • S3 One Zone-IA: exam trap — “cost-optimize infrequently accessed data” → S3-IA; “can tolerate AZ loss” is required for One Zone-IA S3 單一可用區域-IA：考試陷阱——「成本最佳化不常存取的資料」→ S3-IA；選用單一可用區域-IA 的前提是「可承受可用區域損失」

EBS 彈性區塊存放區

  • Tied to a single AZ — cannot directly attach to an instance in another AZ 繫結至單一可用區域——無法直接連接至另一個可用區域的執行個體
  • Snapshots go to S3 (durable, cross-AZ); can be copied cross-region 快照儲存至 S3（持久、跨可用區域）；可跨區域複製
  • io2 Block Express: highest durability (99.999%) and IOPS io2 Block Express：最高持久性（99.999%）和 IOPS

EFS 彈性檔案系統

  • Multi-AZ by default (Regional); automatically accessible from multiple AZs 預設多可用區域（區域性）；可從多個可用區域自動存取
  • EFS One Zone: single AZ, cheaper, less resilient — exam trap for cost questions EFS 單一可用區域：單一可用區域、成本較低、彈性較差——成本問題的考試陷阱

Networking 網路

Elastic IP (EIP) 彈性 IP

  • Static public IP that can be remapped to another instance during failover 可在容錯移轉期間重新對應至另一個執行個體的靜態公有 IP
  • Exam: manual failover pattern = EIP reassignment; automated = Route 53 health checks 考試：手動容錯移轉模式 = EIP 重新指派；自動化 = Route 53 運作狀態檢查

Route 53

  • Failover: active-passive; routes to secondary when primary health check fails 容錯移轉：主動-被動；當主要端點運作狀態檢查失敗時路由至次要端點
  • Latency: routes to lowest-latency region 延遲：路由至延遲最低的區域
  • Weighted: gradual traffic shifting (used in blue/green deployments) 加權：漸進式流量轉移（用於藍/綠部署）
  • Multivalue Answer: returns multiple healthy IPs (client-side load balancing — NOT a replacement for ELB) 多值答案：傳回多個健康 IP（用戶端負載平衡——非 ELB 的替代方案）
  • Health checks can monitor endpoints, other health checks (calculated), and CloudWatch alarms 運作狀態檢查可監控端點、其他運作狀態檢查（計算型）及 CloudWatch 警示

VPC 虛擬私有雲端

  • Subnets are AZ-scoped — always deploy across at least 2 AZs for HA 子網路限定於可用區域——高可用性需跨至少 2 個可用區域部署
  • Internet Gateway is itself HA (AWS-managed) 網際網路閘道本身具有高可用性（由 AWS 管理）

3. Architecture Patterns 架構模式

Pattern 1: Multi-AZ Web Application (Classic HA) 模式 1：多可用區域 Web 應用程式（傳統高可用性）

Services: Route 53 → ALB → ASG (EC2 across 2+ AZs) → RDS Multi-AZ

服務：Route 53 → ALB（應用程式負載平衡器）→ ASG（跨 2 個以上可用區域的 EC2）→ RDS 多可用區域

Why it works: ALB distributes across AZs; ASG replaces failed instances; RDS Multi-AZ handles DB failover automatically.

為何有效：ALB 跨可用區域分配流量；ASG 取代故障執行個體；RDS 多可用區域自動處理資料庫容錯移轉。

Exam signal: “highly available”, “survive AZ failure”, “no single point of failure” → this pattern.

考試信號：「高可用性」、「承受可用區域故障」、「無單點故障」→ 此架構模式。

Pattern 2: Multi-Region Active-Passive DR 模式 2：多區域主動-被動災難復原

Services: Route 53 Failover routing → Primary Region + Secondary Region

服務：Route 53 容錯移轉路由 → 主要區域 + 次要區域

Strategy 策略 RTO RPO Cost 成本
Backup & Restore 備份與還原 Hours 數小時 Hours 數小時 Lowest 最低
Pilot Light 引燃 10s of minutes 數十分鐘 Minutes 數分鐘 Low 低
Warm Standby 暖備援 Minutes 數分鐘 Seconds–minutes 數秒至數分鐘 Medium 中等
Multi-site Active-Active 多站台主動-主動 Near-zero 接近零 Near-zero 接近零 Highest 最高

Pattern 3: Decoupled Architecture with SQS 模式 3：使用 SQS 的解耦架構

Services: Producer → SQS → Consumer ASG

服務：生產者 → SQS（簡單佇列服務）→ 消費者 ASG

Why it works: Queue absorbs traffic spikes; consumers scale independently; if consumers fail, messages stay in queue (not lost).

為何有效：佇列吸收流量激增；消費者獨立擴展；消費者故障時，訊息保留在佇列中（不遺失）。

Exam signal: “component failure should not cause data loss”, “spiky workload”, “decouple producer from consumer” → SQS.

考試信號：「元件故障不應導致資料遺失」、「尖峰工作負載」、「解耦生產者與消費者」→ SQS。

  • Standard queue: at-least-once delivery, best-effort ordering 標準佇列：至少一次交付、盡力排序
  • FIFO queue: exactly-once processing, strict ordering, 3,000 msg/s with batching FIFO 佇列：恰好一次處理、嚴格排序，批次處理可達 3,000 則訊息/秒
  • Visibility timeout default: 30 seconds  |  Max retention: 14 days 可見性逾時預設值：30 秒 | 最長保留期間：14 天

Pattern 4: S3 Static Site with CloudFront 模式 4：搭配 CloudFront 的 S3 靜態網站

Services: S3 (static hosting) + CloudFront + Route 53

服務：S3（靜態託管）+ CloudFront + Route 53

Exam signal: “highly available static website with lowest operational overhead” → S3 + CloudFront.

考試信號：「具最低營運負荷的高可用性靜態網站」→ S3 + CloudFront。

Pattern 5: Aurora Global Database for Global Resilience 模式 5：Aurora 全球資料庫提供全球彈性

Services: Aurora Primary Region → Aurora Global Database → Secondary Region(s)

服務：Aurora 主要區域 → Aurora 全球資料庫 → 次要區域

Exam signal: “global users with low read latency” OR “RPO of seconds, RTO under 1 minute for cross-region DR” → Aurora Global.

考試信號：「全球使用者需要低讀取延遲」或「跨區域 DR 的 RPO 為秒級、RTO 在 1 分鐘內」→ Aurora 全球資料庫。

4. Service Comparisons 服務比較

RDS Multi-AZ vs. RDS Read Replica RDS 多可用區域 vs. RDS 唯讀複本

  • Multi-AZ = automatic failover, standby not readable 多可用區域 = 自動容錯移轉，備用執行個體無法讀取
  • Read Replica = improves read performance, manual failover to standalone 唯讀複本 = 提升讀取效能，手動容錯移轉至獨立執行個體
  • Exam rule: “high availability” → Multi-AZ; “read performance” → Read Replica; both needed → use both 考試規則：「高可用性」→ 多可用區域；「讀取效能」→ 唯讀複本；兩者皆需 → 同時使用

SQS Standard vs. FIFO SQS 標準佇列 vs. FIFO 佇列

  • Standard = higher throughput, possible duplicates, no order guarantee 標準 = 更高輸送量、可能重複、無順序保證
  • FIFO = ordered, exactly-once, lower throughput (3,000/s with batching) FIFO = 有序、恰好一次、較低輸送量（批次處理 3,000/秒）
  • Exam rule: “order matters” or “exactly once” → FIFO; otherwise Standard 考試規則：「順序重要」或「恰好一次」→ FIFO；否則使用標準

ALB vs. NLB

  • ALB = Layer 7, content-based routing, flexible ALB = 第 7 層、內容型路由、彈性高
  • NLB = Layer 4, static IP, extreme performance NLB = 第 4 層、靜態 IP、極高效能
  • Exam rule: “static IP required” or “millions of requests/second” → NLB; HTTP traffic → ALB 考試規則：「需要靜態 IP」或「每秒數百萬個請求」→ NLB；HTTP 流量 → ALB

S3 Standard vs. S3 One Zone-IA S3 標準 vs. S3 單一可用區域-IA

  • Standard = 3+ AZ replication, full durability 標準 = 跨 3 個以上可用區域複寫、完整持久性
  • One Zone-IA = single AZ, 20% cheaper, data lost if AZ fails 單一可用區域-IA = 單一可用區域、便宜 20%、可用區域故障時資料遺失
  • Exam rule: “reproducible data” → One Zone-IA acceptable; “cannot be recreated” → never One Zone-IA 考試規則：「可重新產生的資料」→ 單一可用區域-IA 可接受；「無法重新建立」→ 絕不使用單一可用區域-IA

Aurora vs. RDS

  • Aurora = faster failover (~30s), more replicas (15), Global Database option Aurora = 更快容錯移轉（約 30 秒）、更多複本（15 個）、全球資料庫選項
  • RDS = broader engine support (Oracle, SQL Server) RDS = 更廣泛的資料庫引擎支援（Oracle、SQL Server）
  • Exam rule: “fastest failover” or “cross-region low RPO” → Aurora; “Oracle/SQL Server” → RDS 考試規則：「最快容錯移轉」或「跨區域低 RPO」→ Aurora；「Oracle/SQL Server」→ RDS

Route 53 Failover vs. Multivalue Answer Route 53 容錯移轉 vs. 多值答案

  • Failover = active-passive, requires health checks, clean DR pattern 容錯移轉 = 主動-被動、需要運作狀態檢查、乾淨的 DR 模式
  • Multivalue = returns up to 8 healthy IPs; NOT a load balancer replacement 多值答案 = 最多傳回 8 個健康 IP；非負載平衡器的替代方案
  • Exam rule: “active-passive failover” → Failover routing; don’t confuse Multivalue with ELB 考試規則：「主動-被動容錯移轉」→ 容錯移轉路由；勿將多值答案與 ELB 混淆

5. Exam Traps & Gotchas 考試陷阱與注意事項

  1. RDS Multi-AZ standby is not a read replica. Multi-AZ alone doesn’t help with read scaling. You need read replicas too. RDS 多可用區域備用執行個體不是唯讀複本。單獨使用多可用區域對讀取擴展無幫助，還需要唯讀複本。
  2. EBS snapshots don’t move EBS volumes cross-AZ. Create a snapshot, then create a new volume in the target AZ. EBS itself stays AZ-locked. EBS 快照不會跨可用區域移動 EBS 磁碟區。需建立快照，再於目標可用區域建立新磁碟區。EBS 本身仍鎖定於可用區域。
  3. S3 CRR does not replicate existing objects. Only objects created after replication is enabled. For existing objects → S3 Batch Replication. S3 CRR 不複寫現有物件。僅複寫啟用複寫後建立的物件。現有物件需使用 S3 批次複寫。
  4. EC2 health checks vs. ELB health checks in ASG. Default EC2 health checks only replace terminated/stopped instances. If the app crashes but the instance stays running, configure ELB health checks. ASG 中的 EC2 運作狀態檢查 vs. ELB 運作狀態檢查。預設 EC2 運作狀態檢查僅取代已終止/停止的執行個體。若應用程式當機但執行個體仍在運作，需設定 ELB 運作狀態檢查。
  5. Spot Instances and resilience. Spot is interruptible. “Cost-optimize a fault-tolerant batch job” → Spot is fine. “Never go down” → never pure Spot. Spot 執行個體與彈性。Spot 可被中斷。「成本最佳化容錯批次作業」→ 適合 Spot。「永不停機」→ 絕不使用純 Spot。
  6. Route 53 Multivalue is not a load balancer. Returns multiple IPs; client chooses randomly. No awareness of server load. Route 53 多值答案不是負載平衡器。傳回多個 IP；用戶端隨機選擇。不感知伺服器負載。
  7. Aurora Serverless v1 vs. v2. v1 has cold start latency (scales to zero). v2 scales faster, no cold start. “Sporadic traffic” → Serverless v2. Aurora Serverless v1 vs. v2。v1 有冷啟動延遲（可縮減至零）。v2 擴展更快、無冷啟動。「零星流量」→ Serverless v2。
  8. EFS One Zone vs. EFS Regional. EFS Regional is multi-AZ by default. EFS One Zone loses data if the AZ fails — only use for non-critical data. EFS 單一可用區域 vs. EFS 區域性。EFS 區域性預設為多可用區域。EFS 單一可用區域在可用區域故障時會遺失資料——僅用於非關鍵資料。

7. Practice Questions 練習題

Question 1 — A company runs a three-tier web application on EC2. The database tier uses a single RDS MySQL instance. The application must remain available if a database instance fails, and the read workload is increasing as the user base grows. What should a solutions architect recommend? 問題 1——某公司在 EC2 上執行三層 Web 應用程式。資料庫層使用單一 RDS MySQL 執行個體。若資料庫執行個體故障，應用程式必須維持可用，且隨著使用者增加，讀取工作負載也在增加。解決方案架構師應建議什麼？
  1. Enable RDS Multi-AZ and create a read replica in the same AZ 啟用 RDS 多可用區域並在同一可用區域建立唯讀複本
  2. Enable RDS Multi-AZ and create one or more read replicas 啟用 RDS 多可用區域並建立一個或多個唯讀複本
  3. Create a read replica and configure the application to use it for all queries 建立唯讀複本並設定應用程式對所有查詢使用它
  4. Enable automatic backups and increase the RDS instance size 啟用自動備份並增加 RDS 執行個體大小
Answer: B 答案：B
  • B is correct: Multi-AZ provides automatic failover (HA), and read replicas offload read traffic. These solve different problems and are used together. B 正確：多可用區域提供自動容錯移轉（高可用性），唯讀複本分擔讀取流量。這兩者解決不同問題，應同時使用。

Question 5 — A company’s application uses an Auto Scaling Group behind an ALB. The health check is set to EC2 type. During a recent incident, the application process on an EC2 instance crashed, but the instance remained running. The ALB routed traffic to the crashed instance, causing errors. What should be done to prevent this? 問題 5——某公司的應用程式在 ALB 後方使用 Auto Scaling 群組。運作狀態檢查設定為 EC2 類型。最近一次事故中，EC2 執行個體上的應用程式程序當機，但執行個體仍在運作。ALB 將流量路由至當機的執行個體，導致錯誤。應如何防止此情況？
  1. Enable detailed CloudWatch monitoring on the ASG 在 ASG 上啟用詳細的 CloudWatch 監控
  2. Change the ASG health check type to ELB 將 ASG 運作狀態檢查類型變更為 ELB
  3. Set the ASG cooldown period to 60 seconds 將 ASG 冷卻期間設定為 60 秒
  4. Configure the ALB to use connection draining 設定 ALB 使用連線耗盡
Answer: B 答案：B
  • B is correct: ELB health checks test the application endpoint. When the app crashes but the instance stays up, ELB detects the unhealthy response and the ASG replaces the instance. B 正確：ELB 運作狀態檢查測試應用程式端點。當應用程式當機但執行個體仍在運作時，ELB 偵測到不健康的回應，ASG 便取代該執行個體。
  • A is wrong: CloudWatch monitoring provides metrics visibility but doesn’t trigger instance replacement. A 錯誤：CloudWatch 監控提供指標可見性，但不會觸發執行個體取代。
  • C is wrong: Cooldown controls scaling frequency, not health check behavior. C 錯誤：冷卻控制擴展頻率，而非運作狀態檢查行為。
  • D is wrong: Connection draining handles in-flight requests during scale-in, but doesn’t detect or remove unhealthy instances. D 錯誤：連線耗盡處理縮減期間的進行中請求，但不偵測或移除狀況不佳的執行個體。
